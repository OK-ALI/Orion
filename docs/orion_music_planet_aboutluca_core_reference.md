# Orion Music Planet — Core Reference Brief: Aboutluca / Luca Nardi Portfolio

**Project:** Orion — Music Planet  
**Reference priority:** Highest / Core Reference  
**Core inspiration URL:** https://www.aboutluca.com/  
**Awwwards listing:** https://www.awwwards.com/sites/luca-nardi-portfolio-2  
**Purpose of this document:** Explain why Aboutluca is now the main visual and interaction reference for Orion Music Planet, what should be learned from it, and how Codex should translate that inspiration into an original Orion experience.

---

## 1. Final Decision

Aboutluca / Luca Nardi Portfolio is now confirmed as the **core reference** for Orion Music Planet.

This reference has higher priority than the earlier inspiration sources such as Nuclear, COVEO Music, YUNGBLD Creative Studio, cmd+zest, Planetary, Spotify Wrapped, Radio Garden, Apple Music, and Codrops demos.

Those other references can still be used as secondary notes, but the **main feeling, background behavior, scroll synchronization, cinematic storytelling, and immersive motion direction** should come from Aboutluca.

The goal is not to copy Aboutluca. The goal is to use its **experience logic**:

> A website should not just show information. It should become an immersive journey.

For Orion Music Planet, this becomes:

> Music should not just be a page inside Orion. It should become a living audio universe inside Orion.

---

## 2. What Aboutluca Is About

Aboutluca is the personal portfolio of Luca Nardi, a creative developer and digital designer from Italy. The live website introduces Luca and his work through a highly visual, immersive, interactive journey instead of a normal portfolio grid.

The official Awwwards description calls it an immersive portfolio that blends **WebGL particles, sound, and cinematic storytelling** into a journey through vision, craft, and experience.

Luca also explained the idea behind the site in his own post: a portfolio should do more than list projects; it should tell a story and create something closer to a feeling than a catalogue. He specifically mentioned using **WebGL, particles, typography, and sound** for that emotional experience.

This makes it extremely relevant for Orion Music Planet because Orion Music should also avoid feeling like a normal static music page.

---

## 3. Why This Website Is the Best Reference for Orion Music Planet

Aboutluca works because every part of the interface feels synchronized:

- Background animation
- Scroll movement
- Text reveals
- Particle behavior
- Sound atmosphere
- Chapter-based storytelling
- Cinematic pacing
- Dark visual mood
- Premium typography
- Minimal but intentional foreground UI

The site does not feel like separate sections stacked vertically. It feels like one controlled animated timeline. The user scrolls, but emotionally it feels like moving through a scene.

That exact idea should become the foundation of Orion Music Planet.

Orion Music Planet should not feel like:

```text
Sidebar click → normal page → cards → player
```

It should feel like:

```text
Sidebar click → portal transition → particles awaken → Music Planet forms → scroll moves through a sound universe
```

This is the difference between a feature and an experience.

## 4. Verdict

Aboutluca is the strongest reference because it gives Orion Music Planet the missing emotional layer.

Normal music apps teach layout. Aboutluca teaches **feeling**.

Spotify, Apple Music, Nuclear, and other music apps are useful for structure, but Aboutluca is more important for:

- Immersive background design
- Scroll-driven storytelling
- Particles and WebGL atmosphere
- Smooth chapter transitions
- Emotional pacing
- Sound-supported interaction
- Cinematic reveal style
- “another world inside the app” feeling

For Orion, this means Music Planet should become a separate visual dimension inside the desktop app. When the user enters Music, Orion should feel like it has shifted into a new universe.

**Final verdict:** Aboutluca should be the primary experience reference for Orion Music Planet.

## 5. What Orion Should Take From Aboutluca

### 5.1 Scroll as a Timeline

Scrolling should not only move the content. It should control the entire Music Planet scene.

As the user scrolls:

- Particles should change formation.
- Background depth should shift.
- The virtual camera should move.
- Album planets should appear.
- Artist stars should form constellations.
- Lyrics should appear as floating rings.
- Queue items should align like satellites.
- Recent music should leave subtle trail effects.

The page should feel directed, not random.

### 5.2 Background as Part of the Interface

The background should not be decorative wallpaper. It should act like the living base layer of Music Planet.

The background should react to:

- Current section
- Scroll progress
- Playing / paused state
- Current track intensity
- Album artwork colors
- User hover actions
- Mini player state
- Low GPU / reduce motion settings

### 5.3 Storytelling Through Sections

Aboutluca tells a creative story through chapters. Orion Music Planet should tell a music story through chapters.

Suggested Orion Music Planet story:

1. Enter Music Planet
2. Your Sound Universe
3. Continue Listening
4. Discover Orbits
5. Album Planets
6. Artist Stars
7. Playlist Constellations
8. Now Playing Core
9. Lyrics Rings
10. Queue Satellites
11. Music Memory Trails

Each section should feel like a visual chapter.

### 5.4 Particles as the Main Visual Material

Particles should become Orion Music Planet’s main visual material.

They can represent:

- Stars
- Dust
- Sound waves
- Lyric fragments
- Album glow
- Listening trails
- Playlist constellations
- Music energy
- Transition portals

Particles must be subtle and premium, not noisy.

### 5.5 Sound as Atmosphere

Aboutluca uses sound as part of the feeling. Orion should use music playback as the energy source of the interface.

When music is playing:

- Bass can softly pulse the planet core.
- High frequencies can create tiny spark particles.
- Waveform rings can expand around the album planet.
- Ambient background speed can increase slightly.
- The UI can become warmer or brighter depending on album colors.

When music pauses:

- The scene should calm down.
- Particles slow.
- Glow decreases.
- Orbit rings become softer.

The music should drive the mood without hurting usability.

## 6. Orion Music Planet Vision

Orion Music Planet should be a cinematic, scroll-synced, audio-reactive music world inside Orion.

When the user enters Music Planet, the app should feel like it changes modes completely. The normal Orion interface should fade into a darker, deeper, more atmospheric world. Music should not look like just another sidebar screen. It should feel like a mini universe.

The visual language should be:

- Dark
- Cosmic
- Premium
- Smooth
- Audio-reactive
- Scroll-synced
- Minimal in foreground
- Rich in background
- Emotional but usable
- Cinematic but not heavy

The core metaphor:

```text
Artists = stars
Albums = planets
Songs = moons
Playlists = constellations
Lyrics = orbit rings
Queue = satellites
Recent listening = comet trails
Genres = galaxies
Now playing = planet core
```

This metaphor should guide the UI, background, animations, transitions, and microinteractions.

## 7. Core Experience Flow

### 7.1 Entering Music Planet

When user clicks Music from Orion sidebar:

1. Sidebar Music item glows.
2. Current Orion screen darkens.
3. Background particles begin appearing.
4. A circular Music Planet core forms.
5. The screen softly zooms forward.
6. Text appears briefly: `Entering Music Planet`.
7. The Music Planet home page reveals.

This should feel like a portal, not a normal page navigation.

### 7.2 Music Planet Home

The first screen should show:

- Large title: `Music Planet`
- Subtitle: `Your sound universe inside Orion.`
- Search bar
- Continue Listening
- Recently Played
- Featured Playlists
- Local Library entry
- Now Playing mini core if music is active

The background should be an idle particle space with soft orbit lines.

### 7.3 Scroll Journey

As the user scrolls, the background changes state:

| Section | Background State | UI Feeling |
|---|---|---|
| Intro | Idle space particles | Arrival / calm |
| Library | Galaxy cluster | Personal collection |
| Playlists | Connected constellation lines | Curated mood |
| Albums | Floating planets | Artwork-first browsing |
| Artists | Star clusters | Discovery / identity |
| Now Playing | Central glowing planet | Focus / playback |
| Lyrics | Orbiting text rings | Immersion |
| Queue | Satellite chain | Control / next tracks |
| History | Comet trails | Memory / recents |

### 7.4 Now Playing Core

The Now Playing screen should become the emotional center of Music Planet.

- Album art appears as the central planet.
- Playback ring surrounds the album.
- Progress bar can be circular or orbit-style.
- Lyrics can appear as thin orbiting text bands.
- Queue can appear as small satellites.
- Bass pulse should slightly expand the planet glow.
- Pause state should freeze the orbit gently.

## 8. Suggested Technical Direction

### 8.1 Main Technologies

For a React/Electron/Vite desktop app, use:

| Need | Suggested Tool |
|---|---|
| 3D animated background | Three.js or React Three Fiber |
| Scroll-driven animation | GSAP ScrollTrigger |
| Page transitions | Framer Motion or GSAP |
| Music/audio reactivity | Web Audio API + AnalyserNode |
| Particle systems | Three.js BufferGeometry / shader material |
| Smooth scrolling | Lenis or controlled scroll container |
| Reduced motion | CSS `prefers-reduced-motion` + Orion setting |
| Low GPU fallback | Static gradient/canvas background |

### 8.2 Background Scene Engine

Build a separate `MusicPlanetSceneEngine` instead of scattering random animation code across components.

Suggested scene states:

```ts
export type MusicPlanetSceneState =
  | 'idle-space'
  | 'enter-music-planet'
  | 'library-galaxy'
  | 'playlist-constellation'
  | 'album-orbits'
  | 'artist-stars'
  | 'now-playing-core'
  | 'lyrics-rings'
  | 'queue-satellites'
  | 'memory-trails'
  | 'low-gpu-static';
```

Each state controls:

```ts
interface MusicPlanetSceneConfig {
  particleCount: number;
  particleSpeed: number;
  cameraDepth: number;
  cameraRotation: number;
  bloomStrength: number;
  orbitVisibility: number;
  albumPlanetVisibility: number;
  audioReactivity: number;
  colorPalette: string[];
  backgroundNoise: number;
  transitionDuration: number;
}
```

This gives Codex a clean architecture.

### 8.3 Scroll Sync System

Use scroll progress to update scene state.

Example:

```ts
const sections = [
  { id: 'intro', scene: 'idle-space' },
  { id: 'library', scene: 'library-galaxy' },
  { id: 'playlists', scene: 'playlist-constellation' },
  { id: 'albums', scene: 'album-orbits' },
  { id: 'artists', scene: 'artist-stars' },
  { id: 'now-playing', scene: 'now-playing-core' },
  { id: 'lyrics', scene: 'lyrics-rings' },
  { id: 'queue', scene: 'queue-satellites' },
  { id: 'history', scene: 'memory-trails' },
];
```

When each section becomes active, update the background scene.

### 8.4 Audio Reactivity

Use Web Audio API to analyze playback data.

Possible values:

```ts
interface AudioReactiveData {
  bass: number;
  mids: number;
  highs: number;
  volume: number;
  waveform: Uint8Array;
  frequency: Uint8Array;
}
```

Map audio data carefully:

| Audio Signal | Visual Response |
|---|---|
| Bass | Planet pulse / glow expansion |
| Mids | Orbit movement / wave rings |
| Highs | Small spark particles |
| Volume | Background intensity |
| Pause | Slow idle mode |
| Track change | Color transition |

Keep this subtle. Music Planet should feel alive, not chaotic.

## 10. UX Rules

### 10.1 Motion Must Serve the Interface

Every animation should have a reason:

- Entering Music = transition into another world
- Scroll = chapter movement
- Play = scene becomes alive
- Pause = scene calms
- Hover album = orbit expands
- Add to queue = satellite joins queue
- Like song = small pulse
- Track change = color and planet shift

Avoid random movement.

### 10.2 Performance First

Music Planet must not damage playback or app responsiveness.

Add these settings:

- Reduce Motion
- Low GPU Mode
- Disable Audio Reactive Background
- Static Background
- Particle Density: Low / Medium / High
- Battery Saver Visuals

On weak systems:

- Lower particle count
- Disable bloom
- Disable expensive shaders
- Reduce frame rate if needed
- Replace WebGL scene with static gradient/canvas

### 10.3 Accessibility

Must support:

- Keyboard navigation
- Clear focus states
- Reduced motion
- High contrast enough for controls
- No audio autoplay without user action
- No essential information hidden only in animation
